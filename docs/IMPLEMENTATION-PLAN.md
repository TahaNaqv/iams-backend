# IAMS — Complete Implementation Plan
**From "demo-complete" to "enterprise-grade production-ready"**

**Deployment target:** On-premise
**Scope:** Full — no optional modules cut
**Plan version:** 1.0 (2026-05-12)

---

## Part 1 — Where We Are Today

### Frontend (`iams-frontend/`)
- **State: ~85% feature-complete in mock mode.** 34 routes across 9 modules, 48 shadcn/ui primitives, i18n (EN/AR/FR with RTL), RBAC route gating with 12 permission keys, React Query data layer, service abstraction (mock ↔ API), Zod validation, dark mode, session timeout, keyboard shortcuts. Built with Lovable.
- **Tech:** React 18 + Vite + TS, Tailwind, shadcn/ui, TanStack Query, react-hook-form + Zod, i18next, recharts, vitest.
- **What's missing:** Real backend wiring (currently `VITE_USE_MOCK=true`), test coverage (only 3 real tests), accessibility audit, mobile parity on 2–3 pages, PDF export, real Excel import.

### Backend (`iams-backend/`)
- **State: ~70% feature-complete API.** Django 6.0.2 + DRF + SimpleJWT, 40 models in single `iams` app, 28+ REST resources, JWT auth, custom `HasPermission` RBAC, 5 migrations applied, request-ID tracing, health/ready endpoints, RBAC seed command, file uploads via multipart, throttled login.
- **Tech:** Python 3.12, uv, Django 6, DRF, SimpleJWT, PostgreSQL 16, Docker.
- **What's missing:** Password reset, email/notification dispatch (stored only), MFA, SSO, OpenAPI docs, soft deletes, auto-population of audit trail, PDF/Word report generation, on-prem object storage (local `media/` only), Celery/async, monitoring, CI/CD, gunicorn (currently runs `runserver` in prod), pagination defaults, comprehensive tests.

### Alignment
The frontend service contract (`iams-frontend/docs/api-contract.md`) and backend URL routes are **largely aligned** — most endpoints exist on both sides. The gap is in: (a) payload shape verification, (b) the missing modules (QAIP, CSA, ICFR, real notifications, real audit-trail capture), and (c) production hardening on both sides.

### Requirements Coverage (`IAMS-Requirements-Document.md`)
Of 16 functional modules:
- **8 fully built** (Audit Universe, Planning, Engagement, Findings, CAPs, Dashboard, plus Resource Mgmt, Documents)
- **4 partially built** (UAM, Risk Assessment, Notifications, Audit Trail — UI ready, backend partial)
- **4 not started** (QAIP, CSA, ICFR, Integrations/SSO/ERP)
- **Working Papers** — partial (evidence upload exists, but no versioning, no digital sign-off, no IIA 2330 compliance signals)

## Part 2 — Guiding Principles

1. **Wire before build.** Connect existing FE to existing BE first — uncover real gaps with real data — *then* build missing modules. Don't add features on top of mocks.
2. **Single source of truth for contracts.** Move from prose docs to **OpenAPI schema** generated by `drf-spectacular`, consumed by FE via `openapi-typescript`. Field-name drift dies here.
3. **Production posture from day one.** Gunicorn + nginx + MinIO + CI before more features. The current `runserver`-in-Docker prod is a ticking bomb.
4. **Enterprise = auditable.** Auto-populate the audit trail via DRF signals/middleware, not handwritten calls. The system *audits audits* — its own log integrity is the product.
5. **Async by default for slow paths.** Celery + Redis for email, PDF generation, Excel imports, scheduled reminders, escalations.
6. **No premature features.** Build QAIP/CSA/ICFR only after Phase 2 stabilizes.

## Part 3 — Stack Decisions (On-Premise)

| Reverse proxy / load balancer | nginx (or HAProxy if multi-node) | Battle-tested |
| Container runtime | Docker + docker-compose | Already in use; add k3s as future option |
| Monitoring | **Prometheus + Grafana** | Self-hosted, industry standard |
| Logs | **Loki + Promtail + Grafana** | Same Grafana pane as metrics |
| Tracing | **Jaeger** (or **Tempo** if staying in Grafana stack) | OpenTelemetry-native |
| Error tracking | **Sentry** (self-hosted) | Free if self-hosted |
| Uptime monitoring | **Uptime Kuma** (self-hosted) | Lightweight, alerts to Slack/email |
| CI/CD | **Gitea Actions** or **GitLab CI** (self-hosted) — or GitHub Actions with **self-hosted runners** inside the network | Keeps code & builds on-prem |

### **PHASE 0 — Pre-Flight Hardening** *(1 week)*

Lock the foundations before anyone codes a new feature.

- Add `drf-spectacular` → `/api/schema/`, `/api/docs/`. Wire all ViewSets with `@extend_schema` annotations.
- Add `pytest-django`, `pytest-cov`, `factory-boy`, `freezegun`. Move `tests.py` to `iams/tests/` package. Coverage gate at 70% (≥85% for `views/`, `serializers/`).
- Add `ruff` + `black` + `mypy` (strict on `views/serializers`) + `pre-commit`.
- Configure DRF default pagination (PageNumber, page_size=25, max=200). Field-level filtering via `django-filter` on every list endpoint.
- Replace `runserver` in Dockerfile with `gunicorn` + `uvicorn-worker`. Add `whitenoise` for static, behind nginx in prod.
- Add `redis` service to docker-compose. Add `django-redis` cache backend. Cache `/permissions/`, `/roles/`, `/departments/`, KPI dashboard (30s TTL).
- Soft delete mixin (`is_deleted`, `deleted_at`) + manager + `DELETE` action override → soft on user-facing tables (audits, findings, CAPs, documents).
- Multi-stage Dockerfile, non-root user, `HEALTHCHECK` directive.

**Frontend**
- Generate types from backend OpenAPI: `openapi-typescript` → `src/services/api-types.gen.ts`. Replace hand-rolled `services/models.ts` with generated types + a thin `domain.ts` wrapper for UI-only fields.
- Tighten `tsconfig`: `strict: true`, `noUncheckedIndexedAccess: true`, `noImplicitAny: true`. Fix the resulting errors.
- Add `msw` (Mock Service Worker) and migrate the mock service layer to MSW handlers. Tests run against MSW; dev can still flip to mock via env var.
- Add `eslint-plugin-jsx-a11y` + `vitest-axe`. Wire axe smoke tests into existing page tests.
- Add Vitest tests for `use-data` hooks (mock contracts), `api-errors`, `PermissionRoute` matrix, Dashboard/Findings/Workbook smoke renders.

**Deliverables**
- ✅ Reproducible local stack (`docker compose up` → migrations + seed + ready)
- ✅ OpenAPI schema published
- ✅ Production-shaped Docker image (gunicorn, multi-stage build, non-root user)
- ✅ Pre-commit + CI lint pipeline green

### **PHASE 1 — Backend Integration (Wire the Frontend)** *(2 weeks)*

Goal: flip `VITE_USE_MOCK=false` and have every route work against real data.

**Track 1 — Auth Hardening**
- Add `/auth/password/reset/` and `/auth/password/reset/confirm/` (token-based, email sent via Celery).
- Add `/auth/password/change/` (authenticated, requires current password).
- Add `/auth/me/` PATCH for profile edits.
- JWT lifetimes: access 15min, refresh 7d, rotation on refresh, blacklist on logout. `SIMPLE_JWT['BLACKLIST_AFTER_ROTATION'] = True`.
- Strict CORS allowlist (no `*`), CSRF trusted origins via env.

**Track 2 — Contract Verification**
For each of the 28 endpoints in `iams-frontend/docs/api-contract.md`:
1. Generate FE request from the React Query hook.
2. Hit real backend, diff response shape vs `models.ts`.
3. Fix mismatches **on the backend** (serializer field name/types) unless backwards-incompat for stored data.

Decisions:
- CamelCase JSON via `djangorestframework-camel-case` (FE camelCase, Python snake_case stays internal).
- All dates ISO-8601 with timezone.
- Enum casing aligned exactly (backend already stores `"In Progress"` — keep).

**Track 3 — File Uploads & Evidence**
- Evidence upload: switch from local `media/` to **MinIO** (via `django-storages[boto3]`). Env-driven: dev still uses local, staging/prod uses MinIO.
- Signed URLs (15-min expiry) for downloads instead of streaming through Django.
- File size limit 100MB, ClamAV scan on upload (Celery task; quarantine on virus detection).
- Frontend: progress bar, retry on failure.

**Track 4 — RBAC Wire-Through**
- Verify every endpoint has `HasPermission` with the right key (matches FE's `PermissionRoute`).
- Add `permissions` field to `/auth/me/` response (flat list of permission keys) — FE already reads this.
- Add `iams/tests/test_rbac_matrix.py` that asserts every endpoint × every role returns the expected 200/403.

**Deliverables**
- ✅ FE running entirely against real backend (mock mode becomes optional dev mode)
- ✅ Auth flows complete: login, refresh, logout, password reset, profile edit
- ✅ MinIO storage live
- ✅ ClamAV virus scanning on every upload
- ✅ RBAC matrix test suite green
- ✅ Field-name parity locked

### **PHASE 2 — Audit Trail, Notifications, Workflows** *(2 weeks)*

**Track 1 — Audit Trail (FR-LOG-01..05)**
- Replace ad-hoc `AuditLogEntry.objects.create()` calls with **automatic capture**:
  - Custom DRF `AuditedModelViewSet` mixin → captures create/update/delete on every action, diffs old vs new state, writes to `AuditLogEntry`.
  - GenericForeignKey to the changed object; JSON `changes` field with `{field: {old, new}}`.
  - `actor` from `request.user`, `request_id` from middleware context var.
- Immutability: Postgres role with REVOKE UPDATE/DELETE on `audit_log_entry` for the app user; only `iams_audit_admin` role can purge after retention period.
- Retention: 7-year retention policy via scheduled Celery task. After 1 year, partitions move to compressed archive table; after 7 years, exported to encrypted offsite storage.
- FE: `AuditLog` page already exists — add filters (actor, action, entity type, date range, request_id), full-text search on `changes` JSON.

**Track 2 — Notifications & Email**
- Celery + Redis broker (already in compose from Phase 0).
- Add `django-anymail` with SMTP backend (env-driven; `mailhog` for dev, Postfix relay for prod).
- Notification taxonomy:
  - Audit assigned, audit status change, finding raised, CAP due in 3d, CAP overdue, approval requested, approval approved/rejected, password reset, weekly summary digest, MFA setup reminder.
- Each notification: `Notification` row + email (user pref-gated).
- User preferences (`NotificationPreference` model): per-type opt-in for email/in-app.
- Celery beat schedule: nightly CAP-overdue scan + reminder, weekly digest Mondays 08:00 local, MFA reminders.
- FE: existing `NotificationsTab` becomes a preference editor; existing topbar bell wires to real `/notifications/`, polled every 60s (websocket upgrade in Phase 5).

**Track 3 — Approval Workflow Engine (FR-PLAN-06, FR-ENG-07, FR-CAP-06)**
- Verify `approve`/`reject` actions advance the workflow:
  - On approve: mark current step approved, advance `current_step`, if last → mark request approved, emit signal.
  - On reject: mark request rejected, notify creator.
  - Signal handlers wire to the approved entity (approved audit plan → audits become creatable; approved CAP closure → finding can close).
- Add **Annual Audit Plan approval chain** (FR-PLAN-06): Auditor → Manager → CAE → Board (configurable per org via `ApprovalChainTemplate` model).
- Add **escalation**: if a step is unactioned for N days, escalate to the next role + email manager.
- FE: `WorkflowApprovals` page already exists — wire to real data, surface "my pending approvals" on dashboard.

**Deliverables**
- ✅ Every state change produces an immutable, queryable, request-ID-correlated log entry
- ✅ Email + in-app notification dispatch live, user-preference-gated, scheduled reminders running
- ✅ Approval workflows end-to-end functional with escalation
- ✅ Tests covering signal flows and escalation timing (freezegun)

### **PHASE 5 — Enterprise Hardening** *(2 weeks)*

**Track 1 — Security (FR-UAM-04, FR-WP-08, NFR-Security)**
- **MFA**: `django-otp` + `django-two-factor-auth`. TOTP first; SMS via on-prem gateway if available. Backup codes. MFA enforcement per role (CAE & Admin: required; others: optional, escalate to required after 30 days).
- **Password policy**: configurable complexity (length, classes, history of last 5), brute-force lockout (already throttled — add lockout + admin unlock).
- **Login attempt logging** (FR-UAM-07): every attempt → `LoginAttempt` model with IP, user-agent, request-ID, outcome.
- **Session timeout** (FR-UAM-08): configurable inactivity → refresh-token revocation. FE already has client-side timer; align with backend enforcement.
- **CSP, HSTS, X-Frame-Options, COOP/COEP** headers via `django-csp` + `SecurityMiddleware`.
- **Secrets management**: HashiCorp Vault for SECRET_KEY, DB creds, MinIO keys, SMTP creds. Remove all secrets from `.env` in repo (use `.env.example` as template only).
- **Dependency scanning**: `pip-audit` + `npm audit` in CI. **Trivy** scans on container images.
- **OWASP ZAP** baseline scan in CI against staging.
- **AES-256 at rest** (NFR): MinIO SSE-S3 for files; PostgreSQL TDE via filesystem-level encryption (LUKS) on the DB volume.
- **TLS 1.3** only at nginx; HSTS preload.

**Track 2 — Performance & Scale (NFR-Performance — 500 concurrent users, <3s dashboard, <2s queries)**
- DB indexes: review and add on every `status`, `*_id` FK, `created_at`, `due_date`. `EXPLAIN ANALYZE` on the 10 slowest endpoints.
- DB connection pooling: `pgbouncer` sidecar in compose.
- Query optimization: `select_related`/`prefetch_related` audit on all ViewSets.
- Pagination: enforce on every list endpoint, default 25.
- N+1 detector in tests: `nplusone` package.
- Load test: `locust` script simulating 500 concurrent users on the top 10 endpoints. Target: p95 < 500ms, dashboard < 3s, file upload (10MB) < 15s.
- Static assets cached aggressively by nginx; SPA bundle hash-named for cache-busting.

**Track 3 — Observability**
- **Sentry** (self-hosted) for FE + BE error tracking.
- **Structured JSON logging** to stdout, shipped via Promtail to **Loki**.
- **OpenTelemetry** instrumentation: distributed traces → **Jaeger** (or Tempo).
- **Prometheus metrics**: `django-prometheus`. Custom counters for business events (audits created, findings raised, CAPs overdue, approvals pending).
- **Grafana dashboards**:
  - System: latency p50/p95/p99 by endpoint, error rate, DB connection pool, Celery queue depth.
  - Business: audits in flight by phase, overdue CAP count, weekly finding rate, top auditors by hours.
- **Uptime Kuma** monitoring `/health/` and `/ready/`, alerts to Slack/email/SMS.
- **Alert rules** in Prometheus AlertManager: error rate > 1%, p95 > 1s, DB connections > 80%, Celery queue depth > 1000, disk > 80%.

**Track 4 — CI/CD**
- Self-hosted GitHub Actions runner (or GitLab CI / Gitea Actions) inside the network.
- Pipelines:
  - PR: lint → type check → tests (FE + BE) → build → preview deploy to ephemeral env in dev cluster.
  - Main merge: build images → push to Harbor → Trivy scan → deploy to staging → smoke tests → manual approval → deploy to prod (blue-green via docker-compose with active/inactive swap; nginx upstream switch).
- Replace `deploy.sh` (SSH shell deploy) with the above pipeline. Keep SSH-deploy as the runner's transport but with proper health checks + rollback.
- Database migrations: backward-compat checklist (see `MIGRATION-CHECKLIST.md` to be created), separate migrate step before app deploy, automatic rollback on health-check failure.
- Backup automation: nightly `pg_dump` encrypted with `age` → NAS + offsite via `restic`; quarterly restore drill (calendar reminder).

**Deliverables**
- ✅ MFA enforced for privileged roles, login auditing live
- ✅ p95 < 500ms on top endpoints under 500 concurrent users
- ✅ Sentry + tracing + metrics + alerting wired end-to-end
- ✅ Zero-downtime CI/CD deploy with automated rollback
- ✅ Backup + restore drill executed once

## Part 5 — Key Architecture Decisions (ADRs to write)

| # | Decision | Why |
|---|----------|-----|
| ADR-001 | OpenAPI as single contract source (drf-spectacular → openapi-typescript) | Eliminates field-name drift; one source of truth |
| ADR-002 | CamelCase JSON via DRF renderer, snake_case in Python | FE idiomatic + Python idiomatic |
| ADR-003 | Soft delete on user-facing entities; hard-immutable on logs/audit-trail | Restore-ability vs immutability tradeoff |
| ADR-004 | MinIO + signed URLs (not stream-through Django) | Scale + offload + cost; portable to S3 later |
| ADR-005 | WeasyPrint for canonical reports, Playwright for dashboard exports | Right tool per job |
| ADR-006 | Celery + Redis (not RQ, not Huey) | Mature, observable, escalation timing matters |
| ADR-007 | Auto audit-trail via DRF mixin + GenericFK | Single capture point; impossible to forget |
| ADR-008 | JWT with rotation + blacklist (not pure stateless) | Logout needs to actually log out |
| ADR-009 | DB-level immutability for AuditLogEntry (revoked UPDATE/DELETE for app role) | App bugs cannot corrupt the audit trail |
| ADR-010 | Single `iams` app stays single (not split into sub-apps) | Migration cost > organizational benefit at current size; revisit at 80+ models |
| ADR-011 | Keycloak as SSO front-door, federated to AD via LDAP | One IdP centralizes auth policy; AD users unchanged |
| ADR-012 | Vault for secrets, on-prem | Industry-standard; auditable secret access |
| ADR-013 | Loki+Promtail (not ELK) | Lighter weight; integrates with Grafana already chosen for metrics |

## Part 6 — Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Field-name drift between FE mocks and real BE responses | High | Medium | Phase 0 OpenAPI codegen kills this |
| Phase 3 modules (QAIP/CSA/ICFR) ambiguously specified | Medium | High | Stakeholder workshops *before* coding each; build minimal MVP per module, iterate |
| Auto audit-trail mixin misses edge cases (bulk updates, raw SQL) | Medium | Critical | Pair with DB-level triggers for highest-stakes tables; document bypass paths |
| Celery operational complexity (stuck queues, lost tasks) | Medium | Medium | `acks_late`, idempotent tasks, Flower monitoring, alert on queue depth |
| 7-year audit-log retention growth (FR-LOG-05) | Low | Medium | Partition by year, archive partitions older than 1y to compressed table; export to cold storage at 7y |
| Risk scoring formula changes require historical recomputation | Medium | Low | Version every scoring model; snapshot scores per model version |
| Excel risk workbook formats vary per customer | High | Medium | Mapping config UI; treat workbook schema as user-editable, not hardcoded |
| Local Django dev server still in production (current state) | Critical | Critical | **Phase 0 fixes this — do not defer** |
| SSH-based deploy (current `deploy.sh`) → no rollback, no atomicity | Critical | High | **Phase 5 CI/CD replaces this — do not ship more features until done** |
| Keycloak misconfiguration locks everyone out | Low | Critical | Always keep a local password fallback for emergency admin; test SSO break-glass quarterly |
| MinIO storage failure | Low | High | 3-node MinIO cluster (erasure-coded); nightly snapshot to offsite |
| PostgreSQL primary failure | Low | Critical | Streaming replication to hot standby; documented failover procedure; quarterly drill |
